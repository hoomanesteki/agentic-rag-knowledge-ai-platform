# T1.1 The medallion in dbt

**Theme:** T1 Semantic layer in dbt. **Status:** done.

## What I set out to do

Rebuild the bronze to gold medallion as dbt models, with tests, lineage, and docs, without giving
up the domain-agnostic design. The engine still ingests any domain from its `domain.yaml`, so the
dbt models cannot be hand-written per domain.

## How I did it

dbt models are static SQL, so I generate them from the manifest. `scripts/dbt_codegen.py` reads the
active domain's `domain.yaml` and writes the three layers:

- `models/staging/bronze_<role>.sql`: the raw CSV read as text (lineage).
- `models/silver/silver_<role>.sql`: typed and PII-masked.
- `models/marts/<role>.sql`: the curated gold table the metric layer and the graph read.

The silver transform reuses the exact same column expression as the Python lakehouse builder
(`data/lakehouse._column_expr`), so typing and masking are defined once, not twice. The codegen
also writes `schema.yml` with the tests, deriving foreign-key relationships from the manifest's
graph edges (for example `sales.store_id -> stores.store_id`).

Tests that run on every build:

- schema tests: `not_null` and `unique` on each primary key.
- relationships tests: every graph-edge foreign key must resolve to its parent table.
- a custom governance test, `is_masked`: every column declared as PII must be a masked pseudonym
  in gold, never a raw value.

Everything lands in the `main` schema (a `generate_schema_name` macro pins it), so gold is
queryable as `main.<role>` exactly where the metric layer and graph loader look. dbt writes into
the same DuckDB file the app reads, so there is one database.

## What I tested

- `make dbt-build` for `apparel_ecommerce`: 12 models + 9 data tests, all pass.
- `make dbt-build` for `saas_support`: 12 models + tests, all pass, including
  `is_masked_tickets_customer_email` (the PII column), confirmed `customer_email` is
  `masked:...` in gold.
- The metric layer reads the dbt-built database: `return_rate_by_size(size=M)` returns `0.5`, the
  same governed number as before.
- `make check` stays green (254 tests, eval gate, both domain contracts, no engine-vocab leak). dbt
  is an optional extra, so the offline engine and its tests still install and run without it.
