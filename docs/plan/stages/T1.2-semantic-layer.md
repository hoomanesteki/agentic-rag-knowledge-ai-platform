# T1.2 The semantic layer, parity, and lineage

**Theme:** T1 Semantic layer in dbt. **Status:** done.

## What I set out to do

Close out the data-architecture theme: prove the two build paths cannot drift, name the downstream
consumers so lineage is real, and write down what the semantic layer is.

## How I did it

- **Parity test** (`tests/test_dbt_parity.py`): build the same domain both ways (the Python
  lakehouse builder and `dbt build`) into two DuckDB files, then compare every gold table column by
  column and row by row. It runs dbt as a subprocess so the file handles are clean, and it skips
  when dbt is not installed, so the base test suite stays light. This is the guard that keeps the
  in-app builder and dbt from ever producing different gold.
- **Exposures** (`dbt/models/exposures.yml`, generated by the codegen): the RAG assistant, the
  backoffice dashboard, and the offline eval are declared as dbt exposures that depend on the gold
  marts. Now `dbt docs` answers "what reads this table", so a change to a gold model shows its blast
  radius across the app, the dashboard, and the eval.
- **The semantic layer, written down** (`docs/semantic-layer.md`): `metrics.yaml` is the single
  source of truth for business metrics; the agent, the eval, and the dashboard all read that one
  definition, and it sits on the tested dbt gold tables. Explains why the definitions live in the
  pack (swappable, read at request time) while dbt owns the tables, tests, and lineage.

## What I tested

- The parity test passes: apparel gold from dbt equals gold from the Python builder for all four
  tables.
- `make dbt-build` for apparel: 12 models, data tests (not_null, unique, three relationships
  covering every foreign key, and the is_masked test on any column a pack marks as PII), plus 3
  exposures parsed. All green, no deprecations. Apparel's structured tables declare no PII, so the
  masking test is a ready guard rather than an active one; the customer PII lives in the order
  documents behind the deterministic order gate.
- `make check` stays green (base suite, dbt optional). Lineage docs come from `make dbt-docs`.

## Where the theme stands

T1 is done: the medallion is modeled in dbt with tests and lineage, the semantic layer is defined
once and read everywhere, and a parity test keeps the two build paths honest. Next theme is T2,
richer sample data and knowledge for the domain.
